# 校园快递管理系统前后端项目目录结构设计

## 1. 文档目的

本文档用于帮助你在正式开始编码前，先把整个前后端分离项目的目录结构想清楚。

对新手来说，目录结构非常重要，因为它直接决定：

- 代码会不会越写越乱
- 前端和后端文件放在哪里
- 页面、接口、数据库配置去哪里找
- 后面调试和维护方不方便

本文档采用“课程实践、最小可实现”的思路设计目录结构，不追求复杂，只追求清楚和够用。

## 2. 总体目录建议

我建议你把代码项目单独放在一个新目录中，例如：

```text
d:\软工实践\campus-express-system
```

这个项目根目录下建议拆成三个核心部分：

```text
campus-express-system
├─ frontend
├─ backend
└─ docs
```

说明：

- `frontend` 用来放前端代码
- `backend` 用来放后端代码
- `docs` 用来放和项目实现有关的说明文件

如果你想更简单，也可以只保留：

```text
campus-express-system
├─ frontend
└─ backend
```

但对课程项目来说，保留 `docs` 目录会更规范。

## 4. 前端目录怎么理解

很多新手看到前端目录会慌，其实你只要知道每一层大概干什么就够了。

## 4.1 `frontend/public`

作用：

- 放公共静态资源

比如：

- 网站图标
- 不需要打包处理的静态图片

对当前课程项目来说，这个目录不需要放太多东西。

## 4.2 `frontend/src`

作用：

- 放前端绝大部分业务代码

你可以把它理解成前端项目的核心工作区。

## 4.3 `frontend/src/api`

作用：

- 放所有请求后端的接口代码

比如：

- 登录接口
- 收件接口
- 发件接口
- 异常处理接口

为什么要单独放这里：

- 以后改接口地址更方便
- 页面里不会写一堆重复请求逻辑

## 4.4 `frontend/src/assets`

作用：

- 放项目用到的图片、图标、样式资源

比如：

- logo
- 背景图
- 页面装饰图

## 4.5 `frontend/src/components`

作用：

- 放可复用的小组件

比如：

- 顶部栏
- 侧边菜单
- 表格组件
- 搜索栏

如果你是初学者，前期这个目录可以很少，甚至为空都可以。

## 4.6 `frontend/src/layouts`

作用：

- 放页面整体布局

比如：

- 管理员布局
- 普通用户布局
- 快递员布局

作用是让不同角色页面结构更统一。

## 4.7 `frontend/src/router`

作用：

- 放前端路由配置

通俗地说，就是管理“页面跳转”。

比如：

- 登录后跳到哪个页面
- 普通用户能访问哪些页面
- 管理员能访问哪些页面

## 4.8 `frontend/src/stores`

作用：

- 放前端状态管理代码

简单理解：

- 保存当前登录用户是谁
- 保存当前角色是什么
- 保存登录状态

对课程项目来说，至少会用来存：

- 用户信息
- 角色信息
- token 或登录状态

## 4.9 `frontend/src/utils`

作用：

- 放工具函数

比如：

- 时间格式化
- 状态转换文字
- 本地存储读取

## 4.10 `frontend/src/views`

作用：

- 放所有页面

这是你最常打交道的目录。

为了避免混乱，建议再按角色拆子目录。

### `views/login`

放：

- 登录页

### `views/user`

放：

- 普通用户主页
- 我的快递页面
- 发件申请页面
- 历史记录页面

### `views/courier`

放：

- 快递员主页
- 待处理任务页
- 已处理任务页

### `views/admin`

放：

- 管理员主页
- 收件管理页
- 出库管理页
- 发件管理页
- 异常处理页
- 账号信息页
- 记录查看页

## 5. 后端目录怎么理解

后端目录对新手来说比前端更容易乱，所以更需要提前规范。

## 5.1 `backend/src/main/java`

作用：

- 放后端 Java 主体代码

## 5.2 `CampusExpressApplication.java`

作用：

- 后端入口文件

简单理解：

- 启动后端服务时，程序从这里开始跑

## 5.3 `controller`

作用：

- 放接口路由

比如：

- 登录接口文件
- 收件接口文件
- 发件接口文件
- 异常接口文件

为什么要单独拆这里：

- 这样每一类接口更清楚

## 5.4 `config`

作用：

- 放项目核心配置

比如：

- 配置读取
- 安全配置
- 常量定义

如果你是新手，这里前期只需要放最基本配置就够了。

## 5.5 `mapper`

作用：

- 放 `MyBatis-Plus` 的数据访问接口

比如：

- 用户表 Mapper
- 收件记录 Mapper
- 发件申请 Mapper
- 异常记录 Mapper

## 5.6 `entity`

作用：

- 放数据库实体类

比如：

- 用户表模型
- 收件记录模型
- 发件申请模型
- 异常记录模型

## 5.7 `dto` 和 `vo`

作用：

- 放请求和响应的数据结构

通俗理解：

- 前端发过来的数据是什么样
- 后端返回的数据是什么样

这层能帮助你把接口输入输出写清楚。

说明：

- `dto` 更适合接收前端请求参数
- `vo` 更适合返回给前端展示数据

## 5.8 `service`

作用：

- 放业务逻辑

比如：

- 登录校验逻辑
- 生成取件码逻辑
- 发件申请处理逻辑
- 异常状态更新逻辑

### 为什么这一层特别重要

因为新手最容易把所有逻辑都塞进接口文件里，后面会非常乱。

所以建议尽量做到：

- 接口文件负责接收请求
- service 负责干活

## 5.9 `resources/application.yml`

作用：

- 放后端项目配置

比如：

- 端口配置
- 数据库连接
- `MyBatis-Plus` 配置
- 日志配置

## 5.10 `pom.xml`

作用：

- 记录 Maven 依赖和构建配置

比如：

- `Spring Boot` 依赖
- `MyBatis-Plus` 依赖
- `MySQL Driver`
- 测试依赖

注意：

- 不要随意手改一堆版本号，先以能稳定启动为优先

## 6. docs 目录建议放什么

如果你准备把项目做得更规范，建议保留 `docs` 目录。

可以放：

- 数据库设计说明
- 接口说明
- 演示数据说明
- 开发记录

如果你不想做太复杂，也可以先不建这个目录。

## 7. README 文件建议写什么

项目根目录下建议有一个 `README.md`。

可以简单写：

- 项目名称
- 技术栈
- 前端启动方式
- 后端启动方式
- 数据库说明
- 演示账号

这个文件在答辩和交作业时很有用。

如果你担心页面太多，建议按下面的最小拆法来。

### 普通用户页面

- `UserHomeView.vue`
- `MyExpressView.vue`
- `SendRequestView.vue`
- `UserHistoryView.vue`

### 快递员页面

- `CourierHomeView.vue`
- `CourierTaskView.vue`

### 管理员页面

- `AdminHomeView.vue`
- `ReceiveManageView.vue`
- `OutboundManageView.vue`

## 9. 一个最适合新手的后端接口拆分方案

建议按模块拆文件。

### 登录相关

- `auth.py`

### 用户相关

- `user.py`

### 收件相关

- `receive.py`

### 发件相关

- `send.py`

### 异常相关

- `exception.py`

### 记录查询相关

- `record.py`

这样你以后找代码会很方便。

## 10. 初学者最推荐的目录组织原则

你只要记住下面几条就够了。

### 原则一：页面和接口分开放

不要把请求逻辑直接写得满页面都是。

### 原则二：不同角色页面分开放

不要把普通用户、快递员、管理员页面乱堆在一起。

### 原则三：数据库模型单独放

不要把表结构逻辑塞进后端入口文件。

### 原则四：一个文件只负责一类事情

例如：

- 登录接口一个文件
- 收件接口一个文件
- 异常处理一个文件

### 原则五：先清楚，再高级

课程项目阶段，目录清晰比目录复杂更重要。

## 11. 推荐的真实落地方式

如果你后面让我带着你做，我会建议真正的项目目录最后长这样：

```text
campus-express-system
├─ frontend
├─ backend
├─ docs
└─ README.md
```

其中前端和后端分别独立启动：

- 前端启动后访问浏览器页面
- 后端启动后提供接口
- 数据库存储所有演示数据

这就是一个标准的前后端分离课程项目结构。

## 12. 初学者最常犯的目录问题

### 问题一：所有文件都放在一个文件夹

结果：

- 后面根本找不到代码

### 问题二：目录拆得太细

结果：

- 自己都记不住放哪了

### 问题三：前端页面和接口代码混写

结果：

- 页面越来越乱

### 问题四：后端逻辑全部写在入口文件

结果：

- 一改就全崩

所以对你来说，最合适的是：

**简单但规范**

## 13. 结论

如果你是第一次做前后端分离课程项目，我建议你直接采用本文这套目录结构，不要自己重新发明一套。

因为这套结构有三个优点：

1. 足够简单
2. 足够清楚
3. 足够支撑课程项目

## 14. 一句话总结

**项目目录结构不是越复杂越好，而是要做到前端、后端、页面、接口、数据库配置各有位置，这样你后面开发、调试、联调时才不会乱。**
