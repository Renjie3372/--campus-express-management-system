# 最小数据库设计

## 1. 设计目标

本设计面向课程实践场景，强调最小可实现、最小可演示和最容易落地。

数据库只需要支撑以下业务：

- 登录认证
- 角色主页区分
- 收件入库与用户查件
- 管理员出库
- 发件申请与快递员揽件
- 异常件登记与处理

## 2. 推荐技术选择

建议数据库使用：

- `MySQL 8.0`

原因：

- 常见且稳定
- 容易和前后端项目集成
- 适合课程项目演示和调试

## 3. 最小表结构方案

建议先使用以下 4 张主表：

1. `user`
2. `receive_record`
3. `send_request`
4. `exception_record`

如果后续时间充足，再扩展：

- `pickup_record`
- `operation_log`

## 4. 用户表 `user`

### 4.1 作用

保存系统登录账号与角色信息。

### 4.2 建议字段

| 字段名 | 类型建议 | 说明 |
| --- | --- | --- |
| id | BIGINT | 用户编号 |
| username | VARCHAR(50) | 用户名 |
| account | VARCHAR(50) | 登录账号 |
| password | VARCHAR(100) | 登录密码 |
| role | VARCHAR(20) | 用户角色 |
| create_time | DATETIME | 创建时间 |

### 4.3 角色建议取值

- `user`
- `courier`
- `admin`

### 4.4 预置账号建议

| id | username | account | password | role |
| --- | --- | --- | --- | --- |
| 1 | 普通用户1 | user01 | 123456 | user |
| 2 | 快递员1 | courier01 | 123456 | courier |
| 3 | 管理员1 | admin01 | 123456 | admin |

## 5. 收件记录表 `receive_record`

### 5.1 作用

保存收件流程中的核心数据。

### 5.2 建议字段

| 字段名 | 类型建议 | 说明 |
| --- | --- | --- |
| id | BIGINT | 收件记录编号 |
| tracking_number | VARCHAR(32) | 快递单号 |
| company | VARCHAR(50) | 快递公司 |
| receiver_id | BIGINT | 收件人编号 |
| location | VARCHAR(50) | 存放位置 |
| pickup_code | VARCHAR(20) | 取件码 |
| status | VARCHAR(20) | 当前状态 |
| create_time | DATETIME | 入库时间 |

### 5.3 状态建议

- `待取件`
- `已出库`
- `异常`

如需更细一点，也可以增加：

- `已入库`

## 6. 发件申请表 `send_request`

### 6.1 作用

保存用户发件申请与后续流转状态。

### 6.2 建议字段

| 字段名 | 类型建议 | 说明 |
| --- | --- | --- |
| id | BIGINT | 发件申请编号 |
| sender_id | BIGINT | 发件人编号 |
| receiver_name | VARCHAR(50) | 收件人姓名 |
| receiver_phone | VARCHAR(20) | 收件人电话 |
| receiver_address | VARCHAR(255) | 收件地址 |
| courier_id | BIGINT | 分配的快递员编号 |
| remark | VARCHAR(255) | 备注 |
| status | VARCHAR(20) | 当前状态 |
| create_time | DATETIME | 提交时间 |

### 6.3 状态建议

- `已提交`
- `已分配`
- `已揽件`
- `已完成`
- `异常`

## 7. 异常记录表 `exception_record`

### 7.1 作用

保存异常件登记与处理信息。

### 7.2 建议字段

| 字段名 | 类型建议 | 说明 |
| --- | --- | --- |
| id | BIGINT | 异常编号 |
| tracking_number | VARCHAR(32) | 对应快递单号 |
| exception_type | VARCHAR(50) | 异常类型 |
| description | VARCHAR(255) | 异常说明 |
| result | VARCHAR(255) | 处理结果 |
| status | VARCHAR(20) | 当前状态 |
| create_time | DATETIME | 创建时间 |
| handle_time | DATETIME | 处理时间 |

### 7.3 异常类型建议

- `丢失`
- `损坏`
- `取件信息错误`
- `长时间未取件`
- `其他`

### 7.4 异常状态建议

- `待处理`
- `已处理`

## 8. 表之间的关系

- `receive_record.receiver_id -> user.id`
- `send_request.sender_id -> user.id`
- `send_request.courier_id -> user.id`
- `exception_record.tracking_number` 可通过快递单号与收件记录关联

## 9. 实现建议

- 主键使用 `BIGINT AUTO_INCREMENT`
- 时间字段统一使用 `DATETIME`
- 字段命名统一使用下划线风格
- 所有状态字段统一用 `status`
- 课程演示阶段可先使用预置账号，不必做注册

## 10. 建表顺序建议

1. 创建数据库
2. 创建 `user` 表
3. 创建 `receive_record` 表
4. 创建 `send_request` 表
5. 创建 `exception_record` 表
6. 插入预置账号
7. 插入少量测试数据

## 11. 一句话总结

**对课程实践来说，数据库设计最重要的不是表多，而是能用最少的表把登录、收件、发件和异常处理这几条主流程稳定支撑起来。**
