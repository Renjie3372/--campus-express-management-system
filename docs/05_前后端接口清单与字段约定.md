# 前后端接口清单与字段约定

## 1. 文档目的

本文档用于把“校园快递管理系统”的前后端实现约束尽量固定下来，减少使用 AI 或 `vibe-coding` 开发时常见的偏差问题。

本文件重点解决以下问题：

- 字段名不统一
- 状态值不统一
- 接口路径随意命名
- 前后端请求响应结构不一致
- 页面和接口对不上

## 2. 推荐实现前提

如果课程没有强制要求 Java 后端，本文默认以下技术组合：

- 前端：`Vue 3` + `Vite` + `Vue Router` + `Pinia` + `Axios`
- 后端：`FastAPI` + `SQLAlchemy`
- 数据库：`MySQL 8.0`

如果后端改用 `Spring Boot`，本文中的字段命名、接口路径和返回结构仍建议保持一致。

## 3. 全局约定

### 3.1 命名风格

- 数据库字段使用下划线命名，例如 `pickup_code`
- 后端 JSON 字段也使用下划线命名，例如 `receiver_id`
- 前端表单字段与接口字段保持同名，不额外转换
- 页面展示文案可以用中文，接口字段统一用英文

### 3.2 统一返回结构

所有接口建议统一返回以下 JSON 结构：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

说明：

- `code`：业务状态码，成功时固定为 `200`
- `message`：提示信息
- `data`：返回的实际数据

### 3.3 常用错误返回

```json
{
  "code": 400,
  "message": "参数错误",
  "data": null
}
```

```json
{
  "code": 401,
  "message": "未登录或登录已过期",
  "data": null
}
```

```json
{
  "code": 403,
  "message": "无权限访问",
  "data": null
}
```

```json
{
  "code": 404,
  "message": "记录不存在",
  "data": null
}
```

## 4. 角色与状态枚举

### 4.1 角色枚举

`role` 字段只允许以下取值：

- `user`
- `courier`
- `admin`

### 4.2 收件状态枚举

`receive_record.status` 建议只允许以下取值：

- `待取件`
- `已出库`
- `异常`

### 4.3 发件状态枚举

`send_request.status` 只允许以下取值：

- `已提交`
- `已分配`
- `已揽件`
- `已完成`
- `异常`

### 4.4 异常状态枚举

`exception_record.status` 只允许以下取值：

- `待处理`
- `已处理`

### 4.5 异常类型枚举

`exception_record.exception_type` 建议只允许以下取值：

- `丢失`
- `损坏`
- `取件信息错误`
- `长时间未取件`
- `其他`

## 5. 核心数据对象字段约定

### 5.1 用户对象 `user`

```json
{
  "id": 1,
  "username": "普通用户1",
  "account": "user01",
  "password": "123456",
  "role": "user",
  "create_time": "2026-06-04 10:00:00"
}
```

字段说明：

- `id`：用户编号
- `username`：用户名
- `account`：登录账号
- `password`：登录密码
- `role`：用户角色
- `create_time`：创建时间

### 5.2 收件记录对象 `receive_record`

```json
{
  "id": 1,
  "tracking_number": "YT20260001",
  "company": "圆通",
  "receiver_id": 1,
  "location": "A-01-03",
  "pickup_code": "834521",
  "status": "待取件",
  "create_time": "2026-06-04 10:30:00"
}
```

### 5.3 发件申请对象 `send_request`

```json
{
  "id": 1,
  "sender_id": 1,
  "receiver_name": "张三",
  "receiver_phone": "13800000000",
  "receiver_address": "软件学院宿舍区3栋201",
  "courier_id": 2,
  "remark": "书籍",
  "status": "已分配",
  "create_time": "2026-06-04 11:00:00"
}
```

### 5.4 取件记录对象 `pickup_record`

```json
{
  "id": 1,
  "receive_id": 1,
  "tracking_number": "YT20260001",
  "pickup_code": "834521",
  "pickup_result": "成功",
  "pickup_time": "2026-06-04 10:40:00"
}
```

### 5.5 异常记录对象 `exception_record`

```json
{
  "id": 1,
  "tracking_number": "YT20260002",
  "exception_type": "取件信息错误",
  "description": "用户反映取件码无法匹配",
  "result": "重新核对快递信息并更正取件码",
  "status": "已处理",
  "create_time": "2026-06-04 11:30:00",
  "handle_time": "2026-06-04 12:00:00"
}
```

## 6. 接口清单

以下接口足够支撑课程项目的核心演示。

## 6.1 登录与用户接口

### 1. 用户登录

- 方法：`POST`
- 路径：`/api/auth/login`

请求体：

```json
{
  "account": "user01",
  "password": "123456"
}
```

成功返回：

```json
{
  "code": 200,
  "message": "登录成功",
  "data": {
    "token": "mock-token-user01",
    "user": {
      "id": 1,
      "username": "普通用户1",
      "account": "user01",
      "role": "user"
    }
  }
}
```

### 2. 获取当前登录用户信息

- 方法：`GET`
- 路径：`/api/auth/me`

成功返回：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "id": 1,
    "username": "普通用户1",
    "account": "user01",
    "role": "user"
  }
}
```

### 3. 获取用户列表

- 方法：`GET`
- 路径：`/api/users`
- 权限：`admin`

说明：

- 用于管理员查看账号列表
- 第一版可不做分页，但建议预留 `page`、`page_size`

## 6.2 收件接口

### 1. 新增收件记录

- 方法：`POST`
- 路径：`/api/receive-records`
- 权限：`courier`

请求体：

```json
{
  "tracking_number": "YT20260001",
  "company": "圆通",
  "receiver_id": 1,
  "location": "A-01-03"
}
```

处理规则：

- 后端自动生成 `pickup_code`
- 初始状态写为 `待取件`
- 同一 `tracking_number` 不允许重复新增

成功返回：

```json
{
  "code": 200,
  "message": "收件录入成功",
  "data": {
    "id": 1,
    "tracking_number": "YT20260001",
    "pickup_code": "834521",
    "status": "待取件"
  }
}
```

### 2. 快递员查看自己录入的收件记录

- 方法：`GET`
- 路径：`/api/my/receive-records-created`
- 权限：`courier`

建议查询参数：

- `tracking_number`
- `status`

### 3. 普通用户查看自己的快递

- 方法：`GET`
- 路径：`/api/my/receive-records`
- 权限：`user`

说明：

- 后端根据当前登录用户 `id` 自动查询
- 普通用户不能传别人 `receiver_id`

### 4. 普通用户自助取件

- 方法：`POST`
- 路径：`/api/pickup-records/verify`
- 权限：`user`

请求体：

```json
{
  "tracking_number": "YT20260001",
  "pickup_code": "834521"
}
```

处理规则：

- 只有状态为 `待取件` 的记录允许出库
- 校验成功后更新状态为 `已出库`
- 校验成功后生成一条 `pickup_record`

成功返回：

```json
{
  "code": 200,
  "message": "出库成功",
  "data": {
    "tracking_number": "YT20260001",
    "status": "已出库",
    "pickup_result": "成功"
  }
}
```

## 6.3 发件接口

### 1. 提交发件申请

- 方法：`POST`
- 路径：`/api/send-requests`
- 权限：`user`

请求体：

```json
{
  "receiver_name": "张三",
  "receiver_phone": "13800000000",
  "receiver_address": "软件学院宿舍区3栋201",
  "remark": "书籍"
}
```

处理规则：

- `sender_id` 由后端根据当前登录用户自动写入
- 初始状态写为 `已提交`

### 2. 管理员查看全部发件申请

- 方法：`GET`
- 路径：`/api/send-requests`
- 权限：`admin`

建议查询参数：

- `status`
- `sender_id`

### 3. 普通用户查看自己的发件申请

- 方法：`GET`
- 路径：`/api/my/send-requests`
- 权限：`user`

### 4. 管理员分配快递员

- 方法：`POST`
- 路径：`/api/send-requests/{id}/assign`
- 权限：`admin`

请求体：

```json
{
  "courier_id": 2
}
```

处理规则：

- 更新 `courier_id`
- 同时把状态更新为 `已分配`

### 5. 快递员查看自己的任务

- 方法：`GET`
- 路径：`/api/my/send-tasks`
- 权限：`courier`

说明：

- 仅返回当前快递员的发件任务

### 6. 快递员确认揽件

- 方法：`POST`
- 路径：`/api/send-requests/{id}/pickup`
- 权限：`courier`

处理规则：

- 仅允许操作分配给当前快递员的记录
- 状态更新为 `已揽件`
- 如果你们想在演示时直接结束流程，也可以继续更新到 `已完成`

## 6.4 异常接口

### 1. 新增异常记录

- 方法：`POST`
- 路径：`/api/exception-records`
- 权限：`admin`

请求体：

```json
{
  "tracking_number": "YT20260002",
  "exception_type": "取件信息错误",
  "description": "用户反映取件码无法匹配"
}
```

处理规则：

- 初始状态写为 `待处理`

### 2. 查看异常记录列表

- 方法：`GET`
- 路径：`/api/exception-records`
- 权限：`admin`

建议查询参数：

- `tracking_number`
- `status`

### 3. 处理异常记录

- 方法：`POST`
- 路径：`/api/exception-records/{id}/handle`
- 权限：`admin`

请求体：

```json
{
  "result": "重新核对快递信息并更正取件码",
  "status": "已处理"
}
```

处理规则：

- 写入 `result`
- 更新 `status`
- 写入 `handle_time`

## 6.5 记录查询接口

### 1. 管理员查看系统概览数据

- 方法：`GET`
- 路径：`/api/dashboard/summary`
- 权限：`admin`

建议返回：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "pending_receive_count": 3,
    "send_request_count": 2,
    "exception_count": 1
  }
}
```

### 2. 普通用户查看个人历史记录

- 方法：`GET`
- 路径：`/api/my/history`
- 权限：`user`

说明：

- 第一版可以简单返回“我的收件记录 + 我的发件记录”

## 7. 页面与接口对应关系

### 7.1 登录页

- 调用 `/api/auth/login`
- 登录成功后保存 `token`、`user.id`、`user.role`
- 根据 `role` 跳转不同主页

### 7.2 普通用户主页

- 可调用 `/api/auth/me`
- 可调用 `/api/my/receive-records`
- 可调用 `/api/my/send-requests`

### 7.3 快递员主页

- 可调用 `/api/auth/me`
- 可调用 `/api/my/receive-records-created`
- 可调用 `/api/my/send-tasks`

### 7.4 管理员主页

- 可调用 `/api/auth/me`
- 可调用 `/api/dashboard/summary`

### 7.5 收件管理页

- 调用 `/api/receive-records` 的 `POST`
- 调用 `/api/my/receive-records-created` 的 `GET`

### 7.6 取件页

- 调用 `/api/pickup-records/verify`

### 7.7 发件申请页

- 调用 `/api/send-requests` 的 `POST`

### 7.8 发件管理页

- 调用 `/api/send-requests` 的 `GET`
- 调用 `/api/send-requests/{id}/assign`

### 7.9 快递员任务页

- 调用 `/api/my/send-tasks`
- 调用 `/api/send-requests/{id}/pickup`

### 7.10 异常处理页

- 调用 `/api/exception-records` 的 `POST` 和 `GET`
- 调用 `/api/exception-records/{id}/handle`

## 8. 前端路由建议

建议至少保留以下路由：

- `/login`
- `/user/home`
- `/user/receive-records`
- `/user/pickup`
- `/user/send-requests`
- `/user/history`
- `/courier/home`
- `/courier/receive-records`
- `/courier/tasks`
- `/admin/home`
- `/admin/send-requests`
- `/admin/exception-records`
- `/admin/users`

## 9. AI 协作时的强约束

如果你们要使用 AI 辅助生成代码，建议把以下要求固定写进提示词：

- 不允许擅自新增角色类型
- 不允许擅自修改字段名
- 不允许擅自扩展状态值
- 所有接口返回结构必须使用 `code`、`message`、`data`
- 前端请求字段名必须与本文档保持一致
- 普通用户只能查看自己的数据
- 快递员只能录入收件信息并查看自己录入的收件记录，以及查看分配给自己的任务
- 管理员负责发件分配、异常处理、用户管理和全局查看权限

## 10. 最小验收检查表

在前后端联调后，至少检查以下内容：

1. 登录成功后是否能正确返回 `role`
2. 三类角色是否跳转到不同主页
3. 快递员收件新增后是否生成 `pickup_code`
4. 普通用户是否只能看到自己的快递
5. 普通用户取件成功后状态是否变为 `已出库`
6. 取件成功后是否生成 `pickup_record`
7. 发件申请提交后状态是否为 `已提交`
8. 管理员分配后状态是否变为 `已分配`
9. 快递员确认揽件后状态是否正确更新
10. 异常记录处理后是否写入 `result` 和 `handle_time`

## 11. 一句话总结

**这份文档的核心作用，是把前后端实现中的字段、状态、接口和页面关系先定死，让 AI 生成代码时尽量少跑偏、少返工。**
