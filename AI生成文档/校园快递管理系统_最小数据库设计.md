# 校园快递管理系统最小数据库设计

## 1. 文档目的

本文档用于为课程实践版“校园快递管理系统”提供一套**最小可实现、最小可演示、最容易落地**的数据库设计方案。

这里强调“最小”，意思是：

- 只保留支撑课程演示所需的核心数据表
- 不做复杂企业级建模
- 不追求一步到位
- 优先保证登录、角色主页、收件、发件、异常处理这几条主流程能跑通

本文档适合你这种第一次做前后端分离项目、代码基础一般、需要快速落地课程项目的情况。

## 2. 数据库设计目标

本项目数据库至少要支撑以下业务：

1. 登录认证
2. 不同角色进入不同主页
3. 管理员新增收件记录
4. 普通用户查看自己的快递
5. 管理员完成出库
6. 普通用户提交发件申请
7. 管理员审核并分配快递员
8. 快递员确认揽件
9. 管理员登记和处理异常记录

因此，数据库设计只要能支撑以上流程，就已经足够满足课程实践需要。

## 3. 推荐数据库类型

推荐使用：

- `MySQL 8.0`

原因：

- 常见、稳定、课程项目中最常用
- 前后端分离项目适配度高
- 便于用 `DataGrip` 查看和调试

## 4. 最小表结构方案

为了保证项目简单清楚，我建议最少使用以下 4 张主表：

1. `user`
2. `receive_record`
3. `send_request`
4. `exception_record`

如果你希望演示逻辑更清晰，可以再增加 1 到 2 张辅助表：

- `pickup_record`
- `operation_log`

但第一阶段完全可以先不做。

## 5. 表一：用户表 `user`

### 5.1 作用

用于保存系统登录用户信息和角色信息。

### 5.2 需要支撑的业务

- 登录
- 角色识别
- 不同主页展示
- 收件、发件时关联用户

### 5.3 建议字段

| 字段名 | 类型建议 | 主键 | 非空 | 说明 |
| --- | --- | --- | --- | --- |
| id | BIGINT | 是 | 是 | 用户编号 |
| username | VARCHAR(50) | 否 | 是 | 用户名 |
| account | VARCHAR(50) | 否 | 是 | 登录账号 |
| password | VARCHAR(100) | 否 | 是 | 登录密码 |
| role | VARCHAR(20) | 否 | 是 | 用户角色 |

### 5.4 角色字段建议取值

建议只保留这 3 类：

- `user`
- `courier`
- `admin`

这样后端判断权限和前端判断主页都最简单。

### 5.5 课程演示建议预置数据

| id | username | account | password | role |
| --- | --- | --- | --- | --- |
| 1 | 普通用户1 | user01 | 123456 | user |
| 2 | 快递员1 | courier01 | 123456 | courier |
| 3 | 管理员1 | admin01 | 123456 | admin |

### 5.6 设计说明

- 课程项目阶段可以先明文密码存储，只用于本地演示。
- 如果后面有精力，再改为加密存储。
- 当前不建议先做注册功能，直接预置账号最适合演示。

## 6. 表二：收件记录表 `receive_record`

### 6.1 作用

用于保存收件流程中的核心信息。

### 6.2 需要支撑的业务

- 管理员录入收件信息
- 系统生成取件码
- 普通用户查询自己的快递
- 管理员完成出库

### 6.3 建议字段

| 字段名 | 类型建议 | 主键 | 非空 | 说明 |
| --- | --- | --- | --- | --- |
| id | BIGINT | 是 | 是 | 收件记录编号 |
| tracking_number | VARCHAR(32) | 否 | 是 | 快递单号 |
| company | VARCHAR(50) | 否 | 是 | 快递公司 |
| receiver_id | BIGINT | 否 | 是 | 收件人编号，对应用户表 |
| location | VARCHAR(50) | 否 | 是 | 存放位置 |
| pickup_code | VARCHAR(20) | 否 | 是 | 取件码 |
| status | VARCHAR(20) | 否 | 是 | 当前状态 |
| create_time | DATETIME | 否 | 是 | 入库时间 |

### 6.4 状态建议

建议只保留最简单的几个状态：

- `待取件`
- `已出库`
- `异常`

如果你想更细一点，也可以加：

- `已入库`

但课程项目没有必要搞得太复杂。

### 6.5 设计说明

- `receiver_id` 对应普通用户。
- `pickup_code` 用于用户查看和管理员出库核验。
- 这张表已经基本能支撑整条收件流程。

## 7. 表三：发件申请表 `send_request`

### 7.1 作用

用于保存普通用户提交的发件申请，以及管理员审核和快递员揽件过程。

### 7.2 需要支撑的业务

- 普通用户提交发件申请
- 管理员查看申请
- 管理员分配快递员
- 快递员确认揽件

### 7.3 建议字段

| 字段名 | 类型建议 | 主键 | 非空 | 说明 |
| --- | --- | --- | --- | --- |
| id | BIGINT | 是 | 是 | 发件申请编号 |
| sender_id | BIGINT | 否 | 是 | 发件人编号，对应用户表 |
| receiver_name | VARCHAR(50) | 否 | 是 | 收件人姓名 |
| receiver_phone | VARCHAR(20) | 否 | 是 | 收件人电话 |
| receiver_address | VARCHAR(255) | 否 | 是 | 收件地址 |
| courier_id | BIGINT | 否 | 否 | 分配的快递员编号 |
| remark | VARCHAR(255) | 否 | 否 | 备注 |
| status | VARCHAR(20) | 否 | 是 | 当前状态 |
| create_time | DATETIME | 否 | 是 | 提交时间 |

### 7.4 状态建议

建议只保留：

- `已提交`
- `已分配`
- `已揽件`
- `已完成`
- `异常`

### 7.5 设计说明

- 用户提交申请时，`courier_id` 可以为空。
- 管理员分配后，再写入 `courier_id`。
- 快递员确认揽件后更新状态即可。

## 8. 表四：异常记录表 `exception_record`

### 8.1 作用

用于保存异常件登记与处理信息。

### 8.2 需要支撑的业务

- 管理员新增异常记录
- 管理员查看异常列表
- 管理员处理异常并更新状态

### 8.3 建议字段

| 字段名 | 类型建议 | 主键 | 非空 | 说明 |
| --- | --- | --- | --- | --- |
| id | BIGINT | 是 | 是 | 异常编号 |
| tracking_number | VARCHAR(32) | 否 | 是 | 对应快递单号 |
| exception_type | VARCHAR(50) | 否 | 是 | 异常类型 |
| description | VARCHAR(255) | 否 | 否 | 异常说明 |
| result | VARCHAR(255) | 否 | 否 | 处理结果 |
| status | VARCHAR(20) | 否 | 是 | 当前状态 |
| create_time | DATETIME | 否 | 是 | 创建时间 |
| handle_time | DATETIME | 否 | 否 | 处理时间 |

### 8.4 异常类型建议

建议只做以下几种：

- `丢失`
- `损坏`
- `取件信息错误`
- `长时间未取件`
- `其他`

### 8.5 异常状态建议

建议只保留：

- `待处理`
- `已处理`

### 8.6 设计说明

- 课程实践不需要复杂流程，只要能新增、查看、处理即可。
- `result` 字段用来展示处理内容。

## 9. 可选辅助表一：取件记录表 `pickup_record`

### 9.1 是否必须

不是必须。

如果你的时间紧、代码基础弱，可以先不做。

### 9.2 什么时候值得加

如果你想让“出库流程”在数据库层面更清晰，可以增加此表。

### 9.3 建议字段

| 字段名 | 类型建议 | 说明 |
| --- | --- | --- |
| id | BIGINT | 记录编号 |
| receive_id | BIGINT | 对应收件记录编号 |
| operator_id | BIGINT | 操作管理员编号 |
| pickup_code | VARCHAR(20) | 核验用取件码 |
| status | VARCHAR(20) | 核验结果 |
| create_time | DATETIME | 操作时间 |

## 10. 可选辅助表二：操作日志表 `operation_log`

### 10.1 是否必须

也不是必须。

### 10.2 什么时候值得加

如果你想在答辩时显得更规范，可以加这张表。

### 10.3 建议字段

| 字段名 | 类型建议 | 说明 |
| --- | --- | --- |
| id | BIGINT | 日志编号 |
| operator_id | BIGINT | 操作人编号 |
| operation_type | VARCHAR(50) | 操作类型 |
| detail | VARCHAR(255) | 操作说明 |
| create_time | DATETIME | 操作时间 |

## 11. 表之间的关系

### 11.1 用户表与收件记录表

- 一个普通用户可以对应多条收件记录
- `receive_record.receiver_id -> user.id`

### 11.2 用户表与发件申请表

- 一个普通用户可以提交多条发件申请
- `send_request.sender_id -> user.id`

### 11.3 快递员与发件申请表

- 一个快递员可以处理多条发件申请
- `send_request.courier_id -> user.id`

### 11.4 异常记录与收件记录

- 当前课程实践中，可以只通过 `tracking_number` 关联
- 不强制一定用外键

对初学者来说，先把逻辑做通比一开始把外键全配齐更重要。

## 12. 适合新手的数据库实现建议

### 12.1 主键建议

对初学者来说，可以直接使用：

- `BIGINT AUTO_INCREMENT`

不要一开始就去折腾复杂主键生成方式。

### 12.2 时间字段建议

建议统一使用：

- `DATETIME`

### 12.3 字段命名建议

建议统一使用下划线命名：

- `receiver_id`
- `pickup_code`
- `create_time`

不要一会儿驼峰、一会儿下划线。

### 12.4 状态字段建议

所有状态统一用：

- `status`

这样前后端处理最简单。

## 13. 推荐建库顺序

如果你后面自己动手建数据库，建议按下面顺序来：

1. 建数据库
2. 建 `user` 表
3. 建 `receive_record` 表
4. 建 `send_request` 表
5. 建 `exception_record` 表
6. 插入 3 个预置账号
7. 插入少量测试数据

## 14. 课程实践最小达标数据库标准

如果你只想达到“够做项目、够演示、够答辩”的程度，那么数据库做到下面这样就够了：

- 有 4 张主表
- 能保存 3 类角色用户
- 能保存收件记录
- 能保存发件申请
- 能保存异常记录
- 能支持状态变化

做到这一步，就已经可以很好地支撑当前课程项目。

## 15. 建议结论

对于当前这个项目，我建议你先采用：

- `user`
- `receive_record`
- `send_request`
- `exception_record`

这 4 张主表先做起来。

等你把登录、收件、发件、异常处理跑通以后，如果时间还有富余，再考虑增加：

- `pickup_record`
- `operation_log`

## 16. 一句话总结

**对于课程实践和新手开发来说，数据库设计最重要的不是表多，而是刚好够用、字段清楚、状态统一、能够支撑演示主流程。**
