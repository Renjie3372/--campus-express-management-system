# 本地演示脚本

## 1. 演示目标

本脚本用于指导校园快递管理系统在课堂展示或答辩时的实际操作顺序，目标是让演示过程清晰、稳定、完整。

建议总时长控制在 8 到 12 分钟。

## 2. 演示前准备

演示前建议确认以下内容：

- 系统能在本地正常启动
- 数据库或本地数据文件可正常读取
- 预置账号已存在
- 测试数据已准备完成
- 页面跳转和角色切换正常

### 建议账号

| 角色 | 账号 | 密码 |
| --- | --- | --- |
| 普通用户 | `user01` | `123456` |
| 快递员 | `courier01` | `123456` |
| 管理员 | `admin01` | `123456` |

### 建议测试数据

- 1 条待取件快递
- 1 条待处理发件申请
- 1 条异常记录
- 1 到 2 条历史记录

## 3. 推荐演示顺序

1. 登录成功与失败
2. 不同角色进入不同主页
3. 收件流程
4. 发件流程
5. 异常处理流程
6. 各角色辅助功能展示

## 4. 演示一：登录

### 操作

- 使用 `user01 / 123456` 登录，展示成功登录
- 再用 `user01 / 111111` 登录，展示失败提示

### 预期结果

- 正确密码可进入普通用户主页
- 错误密码不可进入系统
- 页面能给出清晰提示

## 5. 演示二：不同角色主页

### 操作

- 登录普通用户账号，展示“我的快递”“发件申请”“历史记录”入口
- 登录快递员账号，展示“收件入库”“待处理任务”“已处理记录”入口
- 登录管理员账号，展示“发件管理”“异常处理”“用户管理”“系统概览”入口

### 预期结果

- 三类角色主页明显不同
- 每类角色只看到与自己职责相关的功能

## 6. 演示三：收件流程

### 步骤 1：快递员新增收件记录

示例数据：

```text
快递单号：YT20260001
快递公司：圆通
收件人账号：user01
存放位置：A-01-03
```

预期结果：

- 新增成功
- 系统生成取件码
- 状态变为 `待取件`

### 步骤 2：普通用户查看自己的快递

预期结果：

- 能看到刚才录入的快递
- 页面展示快递单号、取件码、存放位置、状态

### 步骤 3：普通用户完成自助取件

预期结果：

- 输入快递单号和取件码后系统核验成功
- 状态更新为 `已出库`
- 系统生成取件记录

## 7. 演示四：发件流程

### 步骤 1：普通用户提交发件申请

示例数据：

```text
寄件人：user01
收件人姓名：张三
收件人电话：13800000000
收件地址：软件学院宿舍区 3 栋 201
备注：书籍
```

预期结果：

- 申请提交成功
- 状态显示为 `已提交`

### 步骤 2：管理员审核并分配快递员

预期结果：

- 管理员能看到该申请
- 成功分配给 `courier01`
- 状态更新为 `已分配`

### 步骤 3：快递员确认揽件

预期结果：

- 快递员能在任务列表中看到该任务
- 点击确认后状态更新为 `已揽件` 或 `已完成`

## 8. 演示五：异常处理流程

### 步骤 1：管理员新增异常记录

示例数据：

```text
快递单号：YT20260002
异常类型：取件信息错误
异常说明：用户反映取件码无法匹配
状态：待处理
```

### 步骤 2：管理员处理异常

示例处理结果：

```text
重新核对快递信息并更正取件码
```

预期结果：

- 异常记录可以新增
- 异常状态可以更新为 `已处理`
- 页面可查看处理结果

## 9. 演示六：辅助功能

建议补充展示：

- 普通用户历史记录页
- 快递员收件记录页或已处理任务页
- 管理员发件记录、异常记录、用户列表或系统概览页

## 10. 现场讲解重点

每一段演示建议围绕以下逻辑来讲：

- 当前角色是谁
- 当前角色正在处理什么业务
- 页面展示了什么关键数据
- 这一步完成后，系统状态发生了什么变化

## 11. 最终达标标准

如果现场能稳定完成以下内容，即可认为项目达到课程展示要求：

- 登录成功与失败都能演示
- 三类角色主页差异清楚
- 收件流程完整
- 发件流程完整
- 异常处理流程完整
- 各角色有辅助页面可展示

## 12. 一句话总结

**本地演示应围绕“登录 -> 角色主页 -> 收件 -> 发件 -> 异常 -> 辅助功能”的顺序展开，重点展示角色、页面和业务流程之间的闭环关系。**
